# 토큰 최적화 전략 요약

img2dwg 프로젝트의 토큰 최적화 전략을 단계별로 정리합니다.

## 🎯 목표

**문제**: 원본 DWG → JSON 변환 시 40만+ 토큰 생성  
**목표**: 60k 토큰 이하로 절감 (OpenAI 파인튜닝 제한)

---

## 📊 최적화 단계별 효과

### 원본 (최적화 없음)

```json
{
  "entities": [
    {
      "type": "line",
      "start": {"x": 1234.56789, "y": 5678.91234},
      "end": {"x": 2345.67891, "y": 6789.12345},
      "layer": "0",
      "color": 256,
      "linetype": "ByLayer"
    }
    // ... 수만 개
  ]
}
```

**토큰 수**: ~400,000 토큰  
**문제**: 파인튜닝 불가능

---

### 1단계: 기본 최적화 (30~40% 절감)

**적용 기법**:
- 좌표 반올림 (소수점 3자리)
- 기본값 제거 (color=256, linetype="ByLayer")
- DXF R2000 다운버전
- 불필요한 레이어 필터링

```json
{
  "entities": [
    {
      "type": "line",
      "start": {"x": 1234.568, "y": 5678.912},
      "end": {"x": 2345.679, "y": 6789.123}
    }
    // ... 기본값 제거
  ]
}
```

**토큰 수**: ~240,000 토큰 (40% 절감)  
**명령어**: `--optimize`

---

### 2단계: RDP + Compact 스키마 (60~85% 절감)

**적용 기법**:
- RDP 알고리즘으로 폴리라인 간소화
- 키 단축 (`"type"` → `"t"`)
- 배열 평탄화 (`[{x,y}]` → `[x,y]`)
- 레이어 테이블화
- 로컬 좌표계

```json
{
  "m": {"f": "file.dwg", "t": "변경"},
  "l": ["0", "WALL", "DOOR"],
  "b": [0, 0, 10000, 10000],
  "e": [
    {"t": "l", "p": [0, 0, 100, 0], "l": 0},
    {"t": "pl", "p": [0,0,100,0,100,50,0,50], "c": 1, "l": 1}
  ]
}
```

**토큰 수**: ~60,000 토큰 (85% 절감)  
**명령어**: `--optimize --compact-schema --rdp-tolerance 1.0`

---

### 3단계: 레이아웃 분석 (95~99% 절감) 🚀

**적용 기법**:
- 고수준 추상화 (LINE → WALL)
- 의미론적 그룹화 (벽, 방, 문, 창문)
- 반복 패턴 감지 및 템플릿화
- 연결선 병합
- 폐곡선 감지

```json
{
  "metadata": {
    "filename": "file.dwg",
    "representation": "layout",
    "compression_ratio": 99.3
  },
  "layout": {
    "walls": [
      {
        "type": "wall",
        "centerline": [[0,0], [10000,0]],
        "thickness": 200,
        "length": 10000
      }
    ],
    "rooms": [
      {
        "type": "room",
        "boundary": [[0,0], [5000,0], [5000,3000], [0,3000]],
        "area": 15.0
      }
    ],
    "openings": [
      {"type": "door", "position": [2500, 0], "width": 900}
    ]
  }
}
```

**토큰 수**: ~4,000~20,000 토큰 (95~99% 절감)  
**명령어**: `--layout-analysis`

---

### 4단계: 이미지 URL 사용 (이미지 토큰 99% 절감) 🚀🚀

**문제**: Base64 인코딩된 이미지가 전체 토큰의 대부분 차지

**Base64 방식**:
```json
{
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": {
            "url": "data:image/jpeg;base64,/9j/4AAQSkZJRg..." 
          }
        }
      ]
    }
  ]
}
```
**이미지 토큰**: ~300,000 토큰

**URL 방식**:
```json
{
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": {
            "url": "https://i.imgur.com/abc123.jpg"
          }
        }
      ]
    }
  ]
}
```
**이미지 토큰**: ~15 토큰

**절감율**: 99.995%  
**명령어**: `--use-image-url --image-service imgur`

## 🎯 최종 권장 조합

### 최고 효율 (권장!)

```bash
# 1. DWG → JSON 변환 (레이아웃 분석)
python scripts/convert_dwg.py \
  --input datas/ \
  --output output/json/ \
  --layout-analysis

# 2. 데이터셋 생성 (이미지 URL)
python scripts/generate_dataset.py \
  --input-json output/json \
  --output output \
  --use-image-url \
  --image-service imgur
```

**최종 토큰 수**:
- JSON 데이터: ~10,000 토큰 (레이아웃 분석)
- 이미지: ~15 토큰 (URL)
- **총합**: ~10,015 토큰

**원본 대비 절감율**: **97.5%** (400,000 → 10,015)

## 🛠️ 구현 상세

### 레이아웃 분석 알고리즘

1. **선 병합** (`_merge_connected_lines`)
   - 연결된 LINE 엔티티 → POLYLINE
   - 각도 허용 오차: 5도
   - 거리 허용 오차: 10 단위

2. **벽 감지** (`_detect_walls`)
   - 평행한 폴리라인 쌍 감지
   - 거리 범위: 100~500mm
   - 중심선 + 두께 계산

3. **방 감지** (`_detect_rooms`)
   - 폐곡선 감지
   - 면적 계산 (m²)
   - 최소 면적: 1m²

4. **개구부 감지** (`_detect_openings`)
   - 90도 호 → 문
   - 크기 범위: 400~1200mm

5. **패턴 감지** (`_detect_patterns`)
   - 동일 면적 방 그룹화
   - 템플릿 + 인스턴스 위치

### 이미지 업로드 서비스

| 서비스 | 장점 | 단점 | 권장 용도 |
|--------|------|------|-----------|
| **Imgur** | 간단, 무료 | 일일 제한 | 소규모 (< 100) |
| **GitHub** | 영구 보관 | 저장소 필요 | 중규모 (100~1000) |
| **Cloudinary** | 최적화 | 설정 복잡 | 대규모 (> 1000) |

## 🎓 학습 포인트

### 왜 레이아웃 분석이 효과적인가?

**저수준 표현**:
- 수천 개의 개별 선 → 각각 JSON 객체
- 중복 정보 (연결된 선의 끝점)
- 의미 없는 세부사항

**고수준 표현**:
- 수십 개의 의미 있는 객체 (벽, 방)
- 중복 제거 (연결선 병합)
- 구조적 정보 (방 크기, 벽 두께)

### 왜 이미지 URL이 효과적인가?

**Base64 인코딩**:
- 바이너리 → 텍스트 변환 (33% 증가)
- 1MB 이미지 → 1.3MB 텍스트
- 모든 문자가 토큰으로 계산

**URL 참조**:
- 짧은 문자열 (50~100자)
- 이미지 데이터는 외부 저장
- OpenAI가 URL에서 이미지 로드
